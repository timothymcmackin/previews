  • Renamed everything to smartpy-tezos as tezos-smartpy was taken on PyPi.

Language ​

  • Added support for sp.is_implicit_account(implements Michelson instruction IS_IMPLICIT_ACCOUNT),

    python
    @sp.module
def main():
    class C(sp.Contract):
        @sp.entrypoint
        def test_is_implicit_account(self, address):
            # Check if an address is an implicit account
            result = sp.is_implicit_account(address)

            match result:
                case Some(key_hash):
                    # Address is an implicit account, key_hash is available
                    sp.send(address, sp.tez(1))
                case None:
                    # Address is a smart contract or smart rollup
                    raise "Address is a smart contract or smart rollup"


sc = sp.test_scenario("test")
c = main.C()
sc += c
alice = sp.test_account("alice")
c.test_is_implicit_account(alice.address, _amount=sp.tez(1))

  • Added pattern matching for list head/tail:

python
import smartpy as sp


@sp.module
def main():
    class ListTestHeadTail(sp.Contract):
        def __init__(self):
            pass

        @sp.private
        def test_head_tail(self):
            lst = [1, 2, 3]
            match lst:
                case []:
                    assert False, "Should not match empty list"
                case [head, *tail]:
                    assert head == 1
                    assert sp.pack(tail) == sp.pack([2, 3])

v0.22.0 ​

Date: 2025-05-09

The release notes are available as a Jupyter notebook.

Features ​

Core ​

  • Added the Rio protocol as the default protocol.
  • The Oxford protocol is now removed.
  • Update to octez-client 22.0.

Language ​

  • Include Python site packages to SmartPy path. Notably that adds the path listed in .pth files and permits to manipulate wheels in editable mode.
  • sp.trace can now be used to trace tickets.
  • The standard library has been moved to a stdlib directory. The preferred way to import it now is now updated. For example: import smartpy.stdlib.list_utils as lu.
  • Expand error map feature to integrate more cases.

Scenario ​

  • Auto-import modules when instantiating a scenario or using a ref.

    The call to sp.test_scenario no longer needs to have the module specified up front. The link between module and scenario is verified when a contract is originated into a scenario. For an example, see test_scenario_context.py.

    The consequence is that inlined modules don't need to be specified during the sp.test_scenario creation.

  • The standard library is available in the scenario under sp.stdlib. No import is needed. For example: scenario.verify(sp.stdlib.utils.mutez_to_nat(sp.mutez(1)) == sp.nat(1))

  • Return origination_result and call_result values to Python.

Documentation ​

  • Added tutorials from docs.tezos.com
  • Updated documentation for type hints
  • Updated documentation for catching exceptions
  • Introduce an overview page that lists all SmartPy types.
  • Introduce a local development page to help with local development with Cursor instructions.
  • Introduce a llm.txt file.
  • Document FA2.1 support.

Bug fixes ​

  • Renamed contract_utils.spy to address_utils.spy and fixed is_kt1 function.

v0.21.0 ​

Date: 2025-03-19

The release notes are available as a Jupyter notebook.

Features ​

Core ​

  • The Quebec protocol is now the default protocol.
  • The Nairobi protocol is now removed.
  • Update to octez-client 21.3.
  • Update to Python 3.10

Language ​

  • Python match/case syntax:

    SmartPy now supports Python 3.10 match/case syntax. For example:

    smartpy
    match action:
    case add_operator(operator):
        self.data.operators[operator] = ()
    case remove_operator(operator):
        del self.data.operators[operator]

  • Type hints:

    The type can now be expressed via Python type hints rather than via sp.cast(). For example:

    smartpy
    def ep(x: sp.int, y: sp.nat): ...

  • New sets syntax: Sets are now written as {x, y, ...} instead of sp.set(x, y, ...). See the manual for details.

  • Add sp.add_seconds and sp.add_days to scenarios, matching the syntax used in contracts.

Scenario ​

  • The scenario.verify function no longer returns scenario object.
  • Added module level lambda apply.

Web IDE ​

  • New Jupyter Notebook web app (running client side) for writing and running SmartPy code and scenarios. See the manual documentation.
  • Added links to Privacy Policy, Terms and Conditions & Cookie Preferences

Documentation ​

  • New documentation site is live.
  • Document real addresses in scenarios see here.
  • Updated to Vitepress 1.6
  • Updated links to Octez docs and Michelson.

Public repository ​

  • Add a public repository for SmartPy releases and Docker images.

Bug fixes ​

  • Can now add module docstrings.
  • Can now define an empty module with Python pass syntax.

v0.19.0 ​

Date: 2024-02-12

Features ​

Core ​

  • The Nairobi protocol is now the default protocol.
  • SmartPy is now a Python library, installable with pip as a wheel. See installation for instructions.
  • Various improvements for error messages: better wording, more consistency, more location info.

Language ​

  • New features:
    • Added sp.timestamp_from_utc.
    • Optional private_ argument for sp.create_contract.
  • Removed the sp. prefix from SmartPy functions whose names coincide with Python built-ins: sum, map, min, max, range. Conversely, trace is now called sp.trace.
  • A new standard library featuring string-integer conversions, rational/fixed-point arithmetic, and more.

Scenario ​

  • More contract syntax in scenarios: sp.cast(x, t), types (sp.int, sp.option[t], etc.), None, sp.Some(x), and sp.variant.<field_name>(<value>).
  • SmartPy errors are now normal Python exceptions and as such can be caught with Python's try/except. This complements the existing _valid and _exception arguments.
  • SmartPy now has a comprehensive system for metadata management: JSON generation, source code uploading, IPFS helpers and source code validity checks.
  • Dynamic contracts: sc.dynamic_contract can now be used to refer to contracts created by other contracts via sp.create_contract.

Web IDE ​

  • Scenario output in GUI: the overall structure has been improved and more context information is included. Addresses now have a copy tooltip.
  • Automatic syntax completion in the IDE has been extended.

Documentation ​

  • A FAQ has been added.
  • More documentation on addresses, contracts, keys and signature, debugging, casting, standard library, metadata.
  • Reorganized documentation on test scenarios.

Breaking changes ​

Templates are now shipped under smartpy.templates and are thus imported by from smartpy.templates import template_name. For an example, see test_import.py.

Modules ​

  • The SmartPy set syntax is now the same as in Python. Non-empty sets are written as {x, y, ...}, whereas set() is used for empty sets.
  • sp.get_and_update argument order has been changed to key, value, map.

Scenario ​

  • sp.test_scenario now takes a name as its first argument. This name is also used as the output directory. Arguments to @sp.add_test have been removed.
  • Contract instantiation now requires a scenario, as a consequence, it must occur after the call to sp.test_scenario.
  • The context arguments are now streamlined into the entrypoint call preceded by an underscore, e.g. c.ep(_sender=..., _now=..., _valid=..., _exception=...). .run() is removed. See Testing contracts.
  • The scenario utils have been renamed from sp.utils to sp.scenario_utils. (The functions in sp.utils are now part of the standard library.)

Deprecated ​

The following language elements have been deprecated.

  • Contract language:
    • trace, in favor of sp.trace
    • sp.sum, sp.map, sp.min, sp.max, sp.range, in favor of sum, map, min, max, range
  • Scenario:
    • sp.set_type_expr, in favor of sp.cast
    • sp.some(x), in favor of sp.Some
    • sp.none, in favor of None
    • TInt, TRecord etc., in favor of sp.int, sp.record etc.
    • Variant creation with sp.variant("<field_name>", <value>) in favor of sp.variant.<field_name>(<value>)

Bug fixes ​

  • Evaluation order of map key assignment: in m[x] = y, y is now evaluated first.
  • Evaluation order of .apply, .get_opt and .contains: m.contains(x), x is now evaluated before m.
  • Fix list comprehension interpretation when calling a lambda with effects.
  • State backtracking in simulator when an expected error fails to occur.

v0.17.0 ​

Date: 2023-03-18

Changelog ​

  • A new frontend with parsed syntax:

    • Control statements no longer have to be prefixed by sp.. You can now write if, for, and while, just like in Python.

    • No need for x = sp.local("x", ...) and x = sp.compute(...) anymore: to assign to a variable, you can now write x = 42, just like in Python.

    • No more x.value to access a variable's value. Just write x instead.

    • Support for list comprehensions: [x+1 for x in xs].

    • In the __init__ function of a contract the storage can now be initialised through assignments, e.g. self.data.x = 42, unlike the old way that required you to use self.init.

      Example:

      smartpy
      @sp.entrypoint
def squareRoot(self, x):
    assert x >= 0
    y = x
    while y * y > x:
        y = (x / y + y) / 2
    assert y * y <= x and x < (y + 1) * (y + 1)
    self.data.value = y

    • Contracts are now part of modules, which are marked by the @sp.module decorator:

      python
      @sp.module
def main():
    class Calculator(sp.Contract):
        def __init__(self):
            self.data.result = 0

        @sp.entrypoint
        def multiply(self, x, y):
            self.data.result = x * y

    • In tests, the module (here main) needs to be given as an argument to sp.test_scenario:

      python
      @sp.add_test()
def test():
    scenario = sp.test_scenario("Calculator", main)
    c = main.Calculator()
    scenario += c

    • For further details, see the new manual.

  • A new deployment mechanism via Docker. This greatly simplifies installation and improves performance. See installation.
